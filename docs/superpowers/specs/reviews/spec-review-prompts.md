```

## 2. 金融领域专家评审

```
你是一位资深基金研究员和CFA持证人，有8年公募基金投研经验，擅长基金业绩评价、投资风格分析和竞品对标。请从专业金融视角评审以下规格文档。

评审重点：
1. 业绩指标是否全面？收益率、夏普比率、最大回撤、Alpha/Beta、信息比率是否覆盖了行业标准？还缺哪些？

7. 全历史回溯的数据范围，在实际分析中是否有必要？哪些历史数据最有分析价值？

## 3. 技术负责人/架构师评审

```
你是一位有15年经验的软件架构师，主导过多个金融数据分析平台的设计与开发。请从技术可行性、架构合理性、可扩展性角度评审以下规格文档。

评审重点：
1. 技术选型：Streamlit作为主框架是否合适？对于"本地单用户"场景是否有更优方案？
2. 分层架构（UI → Service → Data）是否清晰？服务层和数据层的边界是否有问题？
3. SQLite作为存储方案：全历史回溯场景下，性能瓶颈会在哪里？数据量增长后如何应对？
4. 数据模型设计：表结构是否有冗余或缺失？索引设计是否合理？外键约束是否完整？
5. 竞品推荐引擎的算法复杂度：多维度相似度评分在大批量基金（如3000+只）下的性能如何？
6. 报告生成的依赖链：WeasyPrint和python-docx在不同操作系统上的兼容性问题？
7. 错误处理策略是否充分？同花顺API限流、数据不一致、计算异常等场景是否都有兜底方案？
8. 项目目录结构是否合理？是否有职责不清或文件组织问题？
9. 如果未来要支持多用户、云端部署、ETF/LOF扩展，当前架构是否支持？

请指出至少3个技术风险点并给出具体的改进方案。

## 4. 数据工程师评审

```
你是一位有10年经验的数据工程师，专长于金融数据采集、清洗、存储和管道设计。请从数据获取和处理角度评审以下规格文档。

评审重点：
1. 同花顺API对接方案：频率控制、重试策略、缓存机制是否合理？
2. 增量更新策略：如何判断哪些数据需要更新？全量拉取还是增量同步？
3. 数据质量控制：API返回数据如何验证？异常数据如何清洗和标记？
4. 历史数据存储策略：nav_history表每天一条记录，5年数据量约365×3000=100万条，查询性能如何？
5. 数据快照机制：报告生成基于快照，快照的粒度和更新频率如何设计？
6. 如果未来需要接入多数据源（如Wind、东方财富），当前数据层架构是否支持？
7. 相似度计算中使用的收益率曲线相关系数、行业分布等特征工程是否合理？

## 使用建议

1. **优先级**：金融领域专家 > 产品经理 > 技术架构师 > 数据工程师 > UX设计
2. **使用方式**：将规格文档 `docs/superpowers/specs/2026-04-27-fund-competitor-analysis-design.md` 的内容复制到 `<规格文档>` 标签中
3. **评审输出**：每个角色应输出结构化的评审意见，包含问题描述、影响分析、改进建议
